# Jobs-to-Be-Done Framework — Reference

## Core Concept

Users don't buy products — they hire them to do a job. The job is stable over time; solutions change.

## Job Statement Format

```
When [situation/trigger],
I want to [motivation/desired action],
so I can [expected outcome].
```

## Three Layers of Jobs

| Layer | Question | Example |
|-------|----------|---------|
| **Functional** | What task needs to get done? | "Transfer money to a friend" |
| **Emotional** | How do I want to feel? | "Feel confident the money arrived" |
| **Social** | How do I want to be seen? | "Be seen as reliable and prompt" |

## Rules for Good Job Statements

1. **Solution-free**: no product names, features, or technologies
2. **Stable**: the job existed before your product and will exist after
3. **Observable**: you can see someone trying to do this
4. **One job per statement**: compound jobs need splitting
5. **User's language**: not internal jargon

## Common Mistakes

| Mistake | Example | Fix |
|---------|---------|-----|
| Feature as job | "I want to click the export button" | "I want to share this report with my team" |
| Too broad | "I want to manage my finances" | "I want to know if I can afford this purchase right now" |
| Too narrow | "I want to sort by date" | "I want to find recent activity quickly" |
| Company goal as job | "I want to increase engagement" | That's YOUR goal, not the user's |

## Switch Analysis

When users switch from one solution to another, four forces act:

```
PUSH (from current) + PULL (to new) > HABIT (status quo) + ANXIETY (of change)
```

- **Push**: frustration with current solution
- **Pull**: attraction of new solution
- **Habit**: comfort with current workflow
- **Anxiety**: fear of switching (learning curve, data migration, cost)

## Outcome-Driven Innovation (Ulwick)

Jobs have desired outcomes that can be measured:

```
[Direction] + [metric] + [object of control] + [context]
```

Example: "Minimize the time it takes to find a relevant document when preparing for a meeting"

Rate outcomes on: **Importance** (1-10) x **Satisfaction** (1-10)

- High importance + Low satisfaction = **Underserved** (opportunity)
- Low importance + High satisfaction = **Overserved** (cut features)
